# File Conversion

When a user uploads a document or image, most AI models cannot consume the raw file bytes directly. HAWKI therefore runs an automatic **content extraction** step immediately after each file is stored: the file is sent to a configured converter service, which turns it into text and/or images that the model can actually read.

This page explains how that pipeline works, which converter backends are available, and how to extend the setup with a custom converter.

---

## How Conversion Fits Into File Storage

Every time a file is stored through `FileStorageService`, the base class `AbstractFileStorage` calls `ContentExtractor::getExtracts()` right after writing the file to disk. The result — a `FileCollection` of `StoredFileExtract` objects — is then embedded in the `.meta.json` metadata file that lives alongside the original file.

```
{uuid}/
├── {uuid}.blob         ← original file (extension normalized, see below)
├── .meta.json          ← metadata, including references to extracts
└── output/             ← extracted content lives here
    ├── 000-document_content.md
    └── 001-images/image_0_of_page_0.png
```

> **Note on file extensions:** For security reasons, the on-disk extension of the original file is **not** always the real one. Only `pdf`, `doc`, `docx`, `jpg`, `jpeg`, `png`, and `gif` are kept as-is. Every other file type is stored with a `.blob` extension to prevent browsers or servers from executing the file directly if it were ever accessed outside the application. The true original filename and extension are preserved in `.meta.json` and are used for correct handling when the file is served or converted.

When a message references an attachment, HAWKI reads the extracts from `output/` and passes them to the AI model in place of (or alongside) the original binary file.

### Plain Text Short-Circuit

If the uploaded file is already plain text (e.g. `.py`, `.md`, `.json`) and its language type is recognized by `PlainTextLanguageType`, HAWKI skips the external converter entirely. The original file itself is referenced as the extract — no copying or conversion needed.

### When No Converter Is Available

If no converter backend is configured (see below), `ContentExtractor` returns `null` for the extracts. The file is still stored, but no content is extracted, and the AI will not be able to read the document's contents.

## The Conversion Pipeline

```
File upload
    │
    ▼
ContentExtractor::getExtracts()
    │
    ├─ Plain text file? ──► reference original file directly as extract
    │
    ├─ No converter available? ──► return null (no extraction)
    │
    ├─ MIME type mismatch? ──► correct file extension before forwarding
    │
    ├─ canConvertMimetype()? ──► NO → return empty collection
    │
    └─► FileConverterInterface::convert()
            │
            ▼
        FileCollection of FileReference objects
        (markdown, images, OCR text, …)
            │
            ▼
        Written to output/ folder
        Stored as StoredFileExtract objects
```

The `FileConverterInterface` is resolved from Laravel's DI container. The service provider (`FileConverterServiceProvider`) selects the appropriate implementation at boot time based on the configuration (see [Configuration](#configuration) below).

## ImagePreProcessingConverter — Image Format Wrapper

Every configured converter is automatically wrapped by `ImagePreProcessingConverter`. This wrapper transparently extends the underlying converter's capabilities with two additional image format handlers that run entirely on the HAWKI server, without involving the remote conversion service:

| Feature                  | Binary required               | Formats handled                                               |
|--------------------------|-------------------------------|---------------------------------------------------------------|
| **SVG → PNG**            | `rsvg-convert` (librsvg2-bin) | `image/svg+xml`                                               |
| **Exotic images → JPEG** | `convert` (ImageMagick)       | `.ai`, `.eps`, `.ps`, `.psd`, `.tiff`, `.tif`, `.bmp`, `.ico` |

Both binaries are auto-detected at runtime and the result is cached for 24 hours. If neither binary is installed, the wrapper passes all files straight through to the concrete converter unchanged.

The wrapper applies the following logic when a file arrives:

1. If the concrete converter *can* handle this MIME type and *does not* want someone else to handle it → delegate to the concrete converter directly.
2. Otherwise, try `rsvg-convert` for SVGs, then ImageMagick for exotic image types.
3. If a locally converted image (PNG/JPEG) is produced, it is optionally forwarded to the concrete converter as well, so the converter can still extract embedded text via OCR.

> You can use the `FILE_CONVERTER_BINARY_RSVG_CONVERT` and `FILE_CONVERTER_BINARY_IMAGE_MAGICK` environment variables to specify custom paths for these binaries if they are not on the system `PATH`.

## Available Converters

### HawkiDocConverter (`hawki_converter`)

The default converter. Connects to the [HAWKI File Converter](https://github.com/hawk-digital-environments/hawki-toolkit-file-converter) microservice.

- **Supported formats:** PDF, DOC, DOCX
- **Output:** A ZIP archive containing extracted text (Markdown) and embedded images
- **Availability:** Requires `HAWKI_FILE_CONVERTER_API_URL` and `HAWKI_FILE_CONVERTER_API_KEY` to be set

### GwdgDoclingConverter (`gwdg_docling`)

Connects to the GWDG Academic Cloud document conversion endpoint.

- **Supported formats:** Determined by the remote service
- **Output:** Markdown text + base64-encoded images
- **Availability:** Requires `GWDG_API_KEY` and optionally `GWDG_FILE_CONVERTER_API_URL`

### KreuzbergConverter (`kreuzberg`)

Connects to a self-hosted [Kreuzberg](https://github.com/Goldziher/kreuzberg) API instance.

- **Supported formats:** Fetched dynamically from the `/formats` endpoint of the API and cached for 24 hours. Typically covers a broad range of office documents, PDFs, images, and more.
- **Output:** Markdown text + extracted images (with optional OCR via Tesseract, language: `deu+eng`)
- **Availability:** Requires `KREUZBERG_FILE_CONVERTER_API_URL` to be set
- **Note:** Kreuzberg can technically process SVG files, but its output is limited to text extraction. It defers SVG files to the `ImagePreProcessingConverter` wrapper whenever `rsvg-convert` is available, resulting in a much better visual result.

### NullFileConverter (fallback)

Used automatically when no valid converter configuration is found. It always reports `isAvailable() = false`, accepts no MIME types, and produces no extracts. Its presence means the system degrades gracefully — file uploads still work, but document content will not be available to the AI.

## Configuration

Converter selection is controlled by `config/file_converter.php`, driven by environment variables:

```bash
# Select the active converter (default: hawki_converter)
# Available values: hawki_converter, gwdg_docling, kreuzberg
FILE_CONVERTER=hawki_converter

# HAWKI File Converter credentials
HAWKI_FILE_CONVERTER_API_URL=https://your-converter.example.com
HAWKI_FILE_CONVERTER_API_KEY=your-api-key

# GWDG Docling (uses GWDG_API_KEY already set for the AI connection)
GWDG_FILE_CONVERTER_API_URL=https://chat-ai.academiccloud.de/v1/documents/convert

# Kreuzberg (self-hosted)
KREUZBERG_FILE_CONVERTER_API_URL=http://localhost:8000

# Optional: custom paths for image pre-processing binaries
FILE_CONVERTER_BINARY_RSVG_CONVERT=rsvg-convert
FILE_CONVERTER_BINARY_IMAGE_MAGICK=convert
```

The service provider tries the `FILE_CONVERTER` value first, then falls back to `hawki_converter`. If neither produces a valid, available instance, `NullFileConverter` is used.

### Inspecting Supported Types

To see which MIME types (or file extensions) the currently active converter supports, run:

```bash
php artisan filestorage:converter:types:list
# add --extensions to list file extensions instead of MIME types
php artisan filestorage:converter:types:list --extensions
```

## Adding a Custom Converter

To integrate a different conversion backend:

1. **Create a class** that extends `AbstractFileConverter` and implements `FileConverterInterface`. You must provide:
    - `isValidConfig(array $config): bool` — validate that all required config keys are present
    - `isAvailable(): bool` — return `true` if the backend is reachable/usable
    - `getAllowedMimeTypes(): array` — return the list of MIME types your backend accepts
    - `convert(FileReference $file): FileCollection` — perform the conversion and return extracted files

2. **Register it** in `config/file_converter.php`:

   ```php
   'converters' => [
       // ... existing converters ...
       'my_converter' => [
           'class'   => \App\Services\FileConverter\Handlers\MyConverter::class,
           'api_url' => env('MY_CONVERTER_API_URL'),
           // any additional keys your isValidConfig() expects
       ],
   ],
   ```

3. **Activate it** in `.env`:

   ```bash
   FILE_CONVERTER=my_converter
   ```

Your converter will automatically be wrapped by `ImagePreProcessingConverter`, so SVG and exotic image handling is available without any extra work.
